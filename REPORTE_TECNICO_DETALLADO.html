1. RESUMEN EJECUTIVO

SEO Checklist Auditor es un framework automatizado de auditoría SEO técnico construido completamente en Python, que implementa principios avanzados de Programación Orientada a Objetos (POO) siguiendo los principios SOLID, arquitectura en capas limpias y testing exhaustivo mediante pruebas unitarias.

El sistema analiza automáticamente una URL web y evalúa 30 reglas técnicas de SEO basadas en un checklist profesional, generando reportes estructurados en múltiples formatos (JSON, CSV y visual interactivo).

2. PROPÓSITO Y FUNCIONALIDAD DEL PROYECTO

2.1 Problema que Resuelve

En el ámbito de la calidad web (QA) y SEO, validar manualmente las mejores prácticas técnicas de optimización para motores de búsqueda es:
- Repetitivo: Cada sitio requiere revisar decenas de criterios
- Propenso a error humano: Validaciones manuales inconsistentes
- Lento: Puede tomar horas auditar un sitio completo

2.2 Solución Implementada

Este framework automatiza completamente la auditoría de 30 criterios críticos de SEO técnico, incluyendo:

✅ Elementos On-Page: títulos, meta descripciones, headings, canonical
✅ Rendimiento: tiempo de carga, peso de imágenes, TTFB
✅ Arquitectura técnica: robots.txt, sitemap.xml, redirecciones
✅ Marcado estructurado: OpenGraph, Twitter Cards, Schema.org
✅ Accesibilidad SEO: atributos alt, arquitectura de URLs, idioma
✅ Responsive y PWA: viewport meta, manifest, service workers

2.3 Resultados Generados

Para cada regla evaluada, el sistema produce:
- Estado: PASS (cumple), FAIL (falla crítica), WARN (advertencia), MANUAL (requiere revisión humana)
- Detalles: Valores medidos y encontrados
- Recomendación: Acción específica para remediar el problema

5. PRINCIPIOS DE PROGRAMACIÓN ORIENTADA A OBJETOS

5.1 Principios SOLID Implementados

S - Single Responsibility Principle (SRP)

Cada clase tiene una única razón de cambio:

# ✅ CORRECTO: Una clase, una responsabilidad

class BasicUrlNormalizer:  # Solo normaliza URLs
    def normalize(self, value: str) -> str:
        # ... lógica de normalización

class ChecklistRuleEngine:  # Solo evalúa reglas SEO
    def evaluate(self, context: AuditContext) -> list[RuleResult]:
        # ... lógica de evaluación

class RequestsHttpClient:  # Solo realiza peticiones HTTP
    def request(self, method: str, url: str) -> Response:
        # ... lógica de petición

Impacto: Si cambia la forma de normalizar URLs, solo modificas BasicUrlNormalizer sin tocar el motor de reglas o el cliente HTTP.

O - Open/Closed Principle (OCP)

Las clases están abiertas para extensión, cerradas para modificación:

# Puedes crear nuevas estrategias de motor de reglas sin modificar el servicio

class EcommerceRuleEngine(AuditRuleEngine):
    """Motor de reglas específico para e-commerce"""
    def evaluate(self, context: AuditContext) -> list[RuleResult]:
        # Reglas específicas para tiendas online
        return [...]

# Inyectar la nueva estrategia sin cambiar SeoAuditService
service = SeoAuditService(rule_engine=EcommerceRuleEngine())

Impacto: Puedes agregar nuevos tipos de auditorías (e-commerce, SaaS, blogs) sin modificar el código existente.

L - Liskov Substitution Principle (LSP)

Cualquier implementación de una interfaz es intercambiable sin romper el sistema:

# Ambas implementaciones son intercambiables
http_client_1 = RequestsHttpClient(timeout=10)
http_client_2 = AiohttpAsyncClient(timeout=10)  # Hipotético cliente asíncrono

# El servicio funciona con cualquiera sin cambios
service = SeoAuditService(http_client=http_client_1)  # O http_client_2

Impacto: Puedes cambiar de Requests a otro cliente HTTP (httpx, aiohttp) sin tocar la lógica de auditoría.

I - Interface Segregation Principle (ISP)

Interfaces pequeñas y específicas, no monolíticas:

# ❌ INCORRECTO: Interfaz gigante con muchos métodos que no todos necesitan
class MassiveClient(ABC):
    def request(self, ...): ...
    def websocket_connect(self, ...): ...
    def graphql_query(self, ...): ...
    def upload_file(self, ...): ...

# ✅ CORRECTO: Interfaces pequeñas
class HttpClient(ABC):  # Solo HTTP básico
    def request(self, method: str, url: str) -> Any: ...

class UrlNormalizer(ABC):  # Solo normalización
    def normalize(self, value: str) -> str: ...

Impacto: Los clientes solo implementan lo que realmente necesitan, sin métodos inútiles.

D - Dependency Inversion Principle (DIP)

Las clases dependen de abstracciones, no de implementaciones concretas:

# ✅ SeoAuditService NO conoce RequestsHttpClient específicamente
class SeoAuditService:
    def __init__(self, http_client: HttpClient | None = None):  # ← Abstracción
        self.http_client = http_client or RequestsHttpClient()

Impacto: El servicio puede trabajar con cualquier implementación de HttpClient (mock, async, cache) sin modificarse.

5.2 Encapsulación y Ocultamiento de Información

Cada módulo expone solo lo necesario mediante __init__.py:

# seo_auditor/__init__.py - Interfaz pública limpia
from .application.service import SeoAuditService
from .application.serializers import report_to_csv, report_to_json

__all__ = ["SeoAuditService", "report_to_csv", "report_to_json"]

Los detalles internos (modelos, protocolos, excepciones) permanecen ocultos del usuario final.

5.3 Polimorfismo

Múltiples implementaciones de una interfaz pueden usarse indistintamente:

# Polimorfismo en acción
def analyze_with_any_http_client(http_client: HttpClient):
    service = SeoAuditService(http_client=http_client)
    return service.analyze("https://example.com")

# Todas estas llamadas son válidas
result1 = analyze_with_any_http_client(RequestsHttpClient())
result2 = analyze_with_any_http_client(FakeHttpClient())  # Mock para tests
result3 = analyze_with_any_http_client(AsyncHttpClient())  # Hipotético async

5.4 Composición sobre Herencia

El sistema favorece composición (inyección de dependencias) sobre herencia:

# ✅ Composición: SeoAuditService recibe sus dependencias
class SeoAuditService:
    def __init__(
        self,
        http_client: HttpClient,      # ← Compuesto
        normalizer: UrlNormalizer,    # ← Compuesto
        rule_engine: AuditRuleEngine  # ← Compuesto
    ):
        self.http_client = http_client
        self.normalizer = normalizer
        self.rule_engine = rule_engine

6. PATRONES DE DISEÑO IMPLEMENTADOS

6.1 Strategy Pattern (Patrón Estrategia)

Ubicación: ChecklistRuleEngine es una estrategia inyectada en SeoAuditService

Problema resuelto: Permitir diferentes algoritmos de evaluación de reglas sin modificar el servicio.

Implementación:

# Interfaz de estrategia
class AuditRuleEngine(ABC):
    @abstractmethod
    def evaluate(self, context: AuditContext) -> list[RuleResult]:
        raise NotImplementedError

# Estrategia concreta 1: Evaluación estándar
class ChecklistRuleEngine(AuditRuleEngine):
    def evaluate(self, context: AuditContext) -> list[RuleResult]:
        # Aplica 30 reglas SEO estándar
        return [self._rule_1(context), ..., self._rule_30(context)]

# Estrategia concreta 2: Evaluación rápida (hipotético)
class QuickRuleEngine(AuditRuleEngine):
    def evaluate(self, context: AuditContext) -> list[RuleResult]:
        # Solo aplica 5 reglas críticas
        return [self._rule_1(context), ...]

# Inyección de estrategia
service = SeoAuditService(rule_engine=ChecklistRuleEngine())  # O QuickRuleEngine()

Beneficio: Permite crear diferentes perfiles de auditoría (estándar, rápido, específico por industria) sin duplicar código.

6.2 Dependency Injection (Inyección de Dependencias)

Ubicación: Constructor de SeoAuditService

Problema resuelto: Facilita testing y flexibilidad de configuración.

class SeoAuditService:
    def __init__(
        self,
        http_client: HttpClient | None = None,
        normalizer: UrlNormalizer | None = None,
        rule_engine: AuditRuleEngine | None = None,
        sample_size: int = 25,
    ):
        # Valores por defecto si no se inyecta
        self.http_client = http_client or RequestsHttpClient()
        self.normalizer = normalizer or BasicUrlNormalizer()
        self.rule_engine = rule_engine or ChecklistRuleEngine(self.http_client)

Beneficio: En tests unitarios puedes inyectar mocks en lugar de implementaciones reales.

6.3 Facade Pattern (Patrón Fachada)

Ubicación: Clase SEOChecker en seo_checker.py

Problema resuelto: Mantener compatibilidad con API anterior mientras se refactoriza internamente.

class SEOChecker:
    """Facade de compatibilidad para preservar la API pública previa."""

    def __init__(self, timeout: int = 12, sample_size: int = 25):
        self.service = SeoAuditService(
            http_client=RequestsHttpClient(timeout=timeout),
            sample_size=sample_size
        )

    def analyze(self, input_url: str) -> dict[str, Any]:
        return self.service.analyze(input_url)

Beneficio: Los usuarios antiguos del sistema pueden seguir usando SEOChecker sin cambios en su código, mientras internamente se usa la nueva arquitectura.

6.4 Data Transfer Object (DTO) Pattern

Ubicación: @dataclass en models.py

Problema resuelto: Transportar datos estructurados entre capas sin lógica de negocio.

@dataclass
class RuleResult:
    """DTO para resultado de una regla individual"""
    rule_id: int
    item: str
    status: str
    details: str
    recommendation: str
    automated: bool = True

@dataclass
class AuditContext:
    """DTO con todos los datos necesarios para evaluar reglas"""
    input_url: str
    final_url: str
    response: Any
    soup: Any
    # ... 20+ campos más

Beneficio: Estructuras inmutables y type-safe que documentan los datos requeridos.

6.5 Template Method Pattern (Implícito)

Ubicación: Método analyze() en SeoAuditService

Problema resuelto: Define un flujo de ejecución estándar con pasos intercambiables.

def analyze(self, input_url: str) -> dict:
    # 1. Normalizar (delegado a normalizer)
    normalized = self.normalizer.normalize(input_url)

    # 2. Fetch HTTP (delegado a http_client)
    response = self.http_client.request("GET", normalized)

    # 3. Construir contexto (lógica interna)
    context = self._build_context(...)

    # 4. Evaluar reglas (delegado a rule_engine)
    results = self.rule_engine.evaluate(context)

    # 5. Generar reporte (lógica interna)
    return self._build_report(results)

8.2 Flujo Detallado: Desde URL hasta Reporte

PASO 1: Captura de URL (Interfaz)

# En app.py (Streamlit)
url = st.text_input("URL del sitio", placeholder="https://www.ejemplo.com")
if st.button("Analizar"):
    checker = SeoAuditService()
    report = checker.analyze(url)  # ← INICIO DEL FLUJO

PASO 2: Normalización de URL

# En SeoAuditService.analyze()
normalized = self.normalizer.normalize(input_url)
# Input:  "example.com"
# Output: "https://example.com"

# En BasicUrlNormalizer.normalize()
if not re.match(r"^https?://", value, flags=re.I):
    value = "https://" + value  # Agrega protocolo

parsed = urlparse(value)
if not parsed.netloc:  # Valida que existe dominio
    return ""          # Retorna vacío si es inválido

¿Por qué se hace?
- Los usuarios pueden ingresar URLs parciales (sin protocolo)
- Necesitamos URLs absolutas para peticiones HTTP
- Validamos formato antes de intentar descargar

PASO 3: Fetch de Página Principal + Medición de Rendimiento

# Inicio de medición de tiempo
start_time = time.perf_counter()

# Petición HTTP con seguimiento de redirects
response = self.http_client.request("GET", normalized, allow_redirects=True)

# En RequestsHttpClient.request()
try:
    return self.session.request(
        method="GET",
        url="https://example.com",

def _collect_internal_links(soup, base_origin):
    base_host = urlparse(base_origin).netloc  # "example.com"
    links = []
    seen = set()  # Para evitar duplicados

    for a_tag in soup.find_all("a", href=True):
        href = a_tag.get("href", "").strip()

        # Filtrar anchors, mailto, tel, javascript:
        if href.startswith(("#", "mailto:", "tel:", "javascript:")):
            continue

        # Convertir a URL absoluta
        absolute = urljoin(base_origin, href)
        # Ejemplo: href="/contacto" → "https://example.com/contacto"

        parsed = urlparse(absolute)

        # Filtrar solo HTTP/HTTPS del mismo dominio
        if parsed.netloc != base_host:
            continue  # Es link externo

        # Limpiar URL (quitar fragment, normalizar)
        cleaned = urlunparse((parsed.scheme, parsed.netloc, parsed.path or "/", "", parsed.query, ""))

        if cleaned not in seen:
            seen.add(cleaned)
            links.append(cleaned)

    return links

def _collect_images(soup, base_origin):
    images = []
    seen = set()

    for img_tag in soup.find_all("img", src=True):
        src = img_tag.get("src", "").strip()

        # Filtrar data URIs (imágenes base64)
        if src.startswith("data:"):
            continue

        absolute = urljoin(base_origin, src)
        parsed = urlparse(absolute)

        if parsed.scheme in {"http", "https"}:
            clean = urlunparse((parsed.scheme, parsed.netloc, parsed.path, "", "", ""))
            if clean not in seen:
                seen.add(clean)
                images.append(clean)

    return images

PASO 8: Verificación de Recursos Rotos (Broken Links)

def _check_broken_resources(urls, sample_size=25):
    broken = []

    for url in urls[:sample_size]:  # Solo primeras 25 (configurable)
        # Intento 1: HEAD request (más rápido, no descarga contenido)
        response = self.http_client.request("HEAD", url)

        if response is None or response.status_code >= 400:
            # Intento 2: GET request (algunos servidores no soportan HEAD)
            response = self.http_client.request("GET", url)

        if response is None or response.status_code >= 400:
            broken.append(url)  # Confirmado: recurso roto

    return len(broken), broken  # Contador y ejemplos

¿Por qué HEAD primero?
- HEAD no descarga el cuerpo, solo headers (mucho más rápido)
- Algunos servidores mal configurados rechazan HEAD → fallback a GET

PASO 9: Medición de Peso de Imágenes

def _check_image_weight(image_urls, sample_size=25):
    total_kb = 0.0
    heavy = []  # Imágenes >400KB

    for image_url in image_urls[:sample_size]:
        # Intento 1: HEAD para obtener Content-Length
        head_response = self.http_client.request("HEAD", image_url)
        content_len = head_response.headers.get("Content-Length") if head_response else None

        if content_len is None:
            # Intento 2: GET si HEAD no retornó tamaño
            get_response = self.http_client.request("GET", image_url)
            if get_response:
                content_len = get_response.headers.get("Content-Length")

        try:
            size_bytes = int(content_len) if content_len else 0
        except ValueError:
            size_bytes = 0

        size_kb = size_bytes / 1024
        total_kb += size_kb

        if size_kb > 400:  # Umbral: imágenes pesadas >400KB
            heavy.append(image_url)

    return total_kb, heavy

9. MANEJO DE EXCEPCIONES

9.1 Excepciones de Dominio

El sistema define excepciones personalizadas para errores de negocio:

# domain/exceptions.py

class InvalidUrlError(ValueError):
    """Se lanza cuando la URL ingresada no es válida"""
    pass

class FetchError(RuntimeError):
    """Se lanza cuando falla la descarga de la URL principal"""
    pass

¿Por qué excepciones personalizadas?
- Semántica clara: InvalidUrlError es más descriptivo que ValueError genérico
- Permite captura específica: except InvalidUrlError vs except Exception
- Separa errores de dominio de errores técnicos

9.2 Uso de Excepciones en el Flujo

normalized = self.normalizer.normalize(input_url)

    )

response = self.http_client.request("GET", normalized)
if response is None:
    raise FetchError("No fue posible descargar la URL principal.")

9.4 Manejo Silencioso en Infraestructura

El RequestsHttpClient nunca lanza excepciones, siempre retorna None:

# infrastructure/http_client.py
def request(self, method: str, url: str, allow_redirects: bool = True):
    try:
        return self.session.request(method, url, timeout=self.timeout, ...)
    except requests.exceptions.SSLError:
        # Retry sin verificación SSL
        try:
            return self.session.request(..., verify=False)
        except requests.RequestException:
            return None  # ← No lanza, retorna None
    except requests.RequestException:
        return None  # ← No lanza, retorna None

Ventajas:
- El servicio solo verifica if response is None sin try/catch
- Simplifica la lógica de negocio
- Los errores HTTP se manejan como casos de negocio, no excepciones

10.4 Test Doubles: Mock Objects

Problema: El servicio depende de peticiones HTTP reales → tests lentos e inestables

Solución: Crear un Test Double (Fake HTTP Client)

# En test_service.py

class FakeResponse:
    """Simula una respuesta HTTP sin hacer peticiones reales"""
    def __init__(self, url: str, status_code: int, text: str = ""):
        self.url = url
        self.status_code = status_code
        self.text = text
        self.history = []
        self.headers = {}
        self.elapsed = FakeElapsed(0.2)  # TTFB simulado

class FakeHttpClient:
    """Implementa HttpClient pero retorna respuestas pre-programadas"""
    def __init__(self):
        self.responses = {
            ("GET", "https://example.com"): FakeResponse(
                url="https://example.com/",
                status_code=200,
                text="<html>...</html>"  # HTML simulado
            ),
            ("GET", "https://example.com/robots.txt"): FakeResponse(
                url="https://example.com/robots.txt",
                status_code=200,
                text="User-agent: *"
            ),
            # ... más respuestas pre-programadas
        }

    def request(self, method: str, url: str, allow_redirects: bool = True):
        return self.responses.get((method, url))  # lookup directo

Ventajas del Fake:
- ✅ Rápido: Sin latencia de red
- ✅ Determinístico: Siempre retorna lo mismo
- ✅ Sin dependencias externas: No necesita internet
- ✅ Control total: Puedes simular errores específicos

# ✅ Separación de responsabilidades en objetos colaboradores
class SeoAuditService:
    def __init__(self, http_client, normalizer, rule_engine):
        self.http_client = http_client      # Delegación
        self.normalizer = normalizer        # Delegación
        self.rule_engine = rule_engine      # Delegación

    def analyze(self, url):
        normalized = self.normalizer.normalize(url)
        response = self.http_client.request("GET", normalized)
        context = self._build_context(response)
        results = self.rule_engine.evaluate(context)
        return self._build_report(results)

# Cambiar a cliente asíncrono para mayor rendimiento
class AsyncHttpClient(HttpClient):
    async def request(self, method: str, url: str) -> Response:
        async with aiohttp.ClientSession() as session:
            async with session.request(method, url) as response:
                return await response.text()

# Con caché para evitar peticiones duplicadas
class CachedHttpClient(HttpClient):
    def __init__(self, base_client: HttpClient):
        self.base = base_client
        self.cache = {}

    def request(self, method: str, url: str) -> Response:
        cache_key = (method, url)
        if cache_key in self.cache:
            return self.cache[cache_key]

        response = self.base.request(method, url)
        self.cache[cache_key] = response
        return response
